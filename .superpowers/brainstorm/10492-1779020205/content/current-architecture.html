Current Architecture — What We Have

Multi-provider Chrome Extension for archiving AI conversations

Current Structure

src/ — File Layout
types.ts — Shared types (Account, Header, Conversation, ProviderConfig, capabilities)
idb.ts — Dexie IndexedDB (5 tables + helper methods)
common.ts — Shared utils (delay, fetchServiceApi, captureException)
sync-manager.ts — Orchestrates sync across providers
LiveStorage.ts — Reactive storage layer

providers/
  provider-interface.ts — ConversationProvider contract (90 lines)
  provider-registry.ts — Simple Map registry (19 lines)
  gemini/ — 6 files, ~746 lines of working logic
    index.ts, auth.ts, rpc.ts, parser.ts, network.ts, rate-limiter.ts
  openai/ — stub (57 lines)
  claude/ — stub (57 lines)

Pain Points

What's Scattered

  • RPC IDs hardcoded in Gemini (MaZiqc, hNvQHb, etc.) — no central registry
  • Constants (domains, URLs, origins) duplicated across config + implementation
  • Parser logic, auth flows, network handlers split across 6 files per provider
  • No way to test a provider in isolation without loading the full extension
  • Adding a new provider = reading 746 lines of Gemini to understand the pattern
  • Contracts (ProviderConfig, capabilities) defined in types.ts but enforced manually

Goal

One central control layer where functions, constants, methods, and contracts are registered and managed — enabling:

  • Quicker development — new provider = fill in a template, not reverse-engineer Gemini
  • Easier testing — unit test each provider method in isolation
  • Easy onboarding — add providers, methods, protocol types without touching core logic
